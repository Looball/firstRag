# 开发设计

## assistant.py 审查对象

审查文件：`assistant.py`

当前职责：

- 初始化本地 Chroma 向量库检索器。
- 使用 `ChatDeepSeek`、提示词模板和 LCEL 组装 RAG 问答链。
- 通过 Streamlit 展示聊天界面。
- 使用 `st.session_state` 保存会话消息和链对象。
- 将模型回答以手动流式渲染方式输出到页面。

## 当前代码整体评价

`assistant.py` 已经具备一个最小可用 RAG 聊天 Demo 的完整闭环：用户输入、问题改写、向量检索、上下文拼接、模型回答、页面展示都已经连起来了。

下一阶段最重要的目标不是继续堆功能，而是先把稳定性、可观测性和代码边界补起来。这样后面增加文档上传、检索片段展示、参数调节和评估功能时，代码不会很快变得难维护。

## 需要优先处理的问题

### 1. 当前用户问题被重复传入模型

位置：`assistant.py:192-203`

当前逻辑是在生成回答前先执行：

```python
st.session_state.messages.append(("human", prompt))
```

随后又把 `input=prompt` 和包含当前 `prompt` 的 `chat_history` 一起传给链。这样提示词里会出现两份当前用户问题：

- 一份来自 `chat_history`。
- 一份来自 `("human", "{input}")`。

影响：

- 第一次提问时，`chat_history` 已经非空，会进入问题改写链，额外调用一次 LLM。
- 当前问题被重复放进 prompt，增加 token 成本。
- 多轮对话时，问题改写和最终回答可能受到重复消息干扰。

建议改法：

```python
history_for_chain = st.session_state.messages.copy()

st.session_state.messages.append(("human", prompt))

answer = get_answer(
    chain=st.session_state.qa_history_chain,
    input=prompt,
    chat_history=history_for_chain,
)
```

也就是：页面可以立即显示当前用户输入，但传给链的历史记录应当是不包含当前问题的旧历史。

### 2. 缺少环境和向量库状态检查

位置：`assistant.py:24-32`、`assistant.py:52-60`

当前代码直接读取：

- `DEEPSEEK_API_KEY`
- `ZAI_EMD_API`
- `./vector_db/chroma`

如果 API Key 不存在、向量库目录不存在、向量库为空，页面可能在加载阶段直接报错。

建议新增启动检查：

- 检查 `DEEPSEEK_API_KEY` 是否存在。
- 检查 `ZAI_EMD_API` 是否存在。
- 检查 `vector_db/chroma` 是否存在。
- 检查 Chroma collection 是否有数据。
- 在 Streamlit 页面用 `st.error` 或 `st.warning` 给出清晰提示。

### 3. 缺少用户可见的错误处理

位置：`assistant.py:199-208`

当前模型调用、检索、流式输出都没有 `try/except`。一旦 API 请求失败、网络失败、Key 失效、向量库损坏，用户会看到完整 traceback。

建议：

- 给 `render_stream(answer)` 外层增加异常捕获。
- 页面提示“生成失败，请检查 API Key、网络或向量库状态”。
- 终端或日志里保留详细错误，页面上只展示友好的摘要。

### 4. 检索结果不可见

位置：`assistant.py:91-132`

当前用户只能看到最终回答，看不到模型参考了哪些文档片段。这不利于学习 RAG，也不利于判断回答是否可靠。

建议下一步增加：

- 展示检索到的原文片段。
- 展示来源文件名、页码、chunk 序号等 metadata。
- 用 `st.expander("查看检索依据")` 折叠展示。
- 如果没有检索到内容，明确提示“知识库中没有找到相关片段”。

### 5. 配置硬编码较多

位置：`assistant.py:24-32`、`assistant.py:53-58`、`assistant.py:102-108`

当前这些值直接写死在代码中：

- 向量库路径：`./vector_db/chroma`
- 检索数量：`k=2`
- 模型名：`deepseek-v4-flash`
- temperature：`0.2`
- max_tokens：`8000`
- 系统 prompt

建议把它们集中到配置区，或拆成 `config.py`：

```python
PERSIST_DIRECTORY = "./vector_db/chroma"
RETRIEVAL_K = 2
MODEL_NAME = "deepseek-v4-flash"
TEMPERATURE = 0.2
MAX_TOKENS = 8000
```

后续也可以在 Streamlit sidebar 中开放部分参数。

## 代码结构提升方向

### 第一阶段：保持单文件，但先变清晰

适合下一次小改。

- 把所有 import 尽量放到文件顶部，保留少数必要的延迟 import。
- 增加配置常量区。
- 增加 `validate_environment()`。
- 增加 `load_vectorstore()` 或 `get_vectorstore()`。
- 修复 chat history 重复传入问题。
- 增加 `try/except` 包住模型生成。
- 增加“清空对话”按钮。

建议目标：`assistant.py` 仍然是单文件，但主流程更稳定。

### 第二阶段：拆分模块

适合功能开始变多时做。

建议拆分为：

```text
assistant.py       # Streamlit UI 入口
rag_chain.py       # get_retriever / get_chain / get_answer
config.py          # 路径、模型、检索参数
prompts.py         # 问题改写 prompt、回答 prompt
utils.py           # 环境检查、格式化文档、错误处理
```

拆分后收益：

- UI 和 RAG 逻辑分离。
- 以后可以复用同一套 RAG 链做 CLI、API 或测试。
- prompt 修改不会干扰页面逻辑。

### 第三阶段：引入测试

适合项目进入稳定迭代后做。

优先测试：

- `get_res_doc()` 能正确拼接文档内容。
- 环境变量缺失时返回清晰错误。
- 空检索结果时不会崩溃。
- chat history 不包含当前输入的重复消息。
- prompt 组装后包含预期字段。

## 功能升级路线图

### v0.1 稳定版：让现有聊天可靠

目标：当前 Streamlit RAG 页面能稳定运行，不再因为配置或环境问题直接崩溃。

功能清单：

- 修复当前问题重复进入 `chat_history` 的问题。
- 页面启动时显示知识库状态。
- 增加 API Key 缺失提示。
- 增加异常处理和友好错误提示。
- 增加清空对话按钮。
- 增加 sidebar，展示当前模型、检索数量、向量库路径。

### v0.2 可解释版：让 RAG 过程可见

目标：用户不仅看到回答，还能看到回答依据。

功能清单：

- 展示检索到的文档片段。
- 展示文档来源、页码、metadata。
- 展示本轮补全后的独立问题。
- 支持调整 `k` 值。
- 当没有相关文档时，引导用户重建知识库或换一种问法。

### v0.3 知识库管理版：让文档更新更方便

目标：从“命令行建库”升级到“页面内管理知识库”。

功能清单：

- 页面上传 PDF 或 Markdown。
- 页面触发文档切分和向量库重建。
- 展示已入库文件列表。
- 展示 chunk 数量、更新时间、向量库路径。
- 支持清空并重建向量库。

### v0.4 检索效果评估版：让学习更系统

目标：从能用升级到能比较、能评估。

功能清单：

- 保存用户问题、检索片段、回答结果。
- 支持不同 `k` 值对比。
- 支持不同 chunk size / overlap 对比。
- 增加“答案是否有依据”的人工标注。
- 整理一个简单评估表格，观察 RAG 参数对结果的影响。

### v0.5 产品化雏形：让体验更完整

目标：从学习 Demo 变成一个小型本地知识库助手。

功能清单：

- 多会话历史。
- 会话保存和恢复。
- 多模型切换。
- 回答中带引用编号。
- 支持导出问答记录。
- 支持只基于知识库回答或允许模型补充回答两种模式。

## 推荐下一步实施顺序

优先顺序建议如下：

1. 修复 `chat_history` 重复传入问题。
2. 增加环境变量和向量库检查。
3. 增加错误处理。
4. 增加清空对话按钮。
5. 展示检索片段和来源。
6. 把配置常量集中管理。
7. 再考虑拆分模块。

这个顺序的原因是：先减少隐藏 bug 和运行失败，再提高 RAG 的可解释性，最后再拆结构。这样每一步都能直接提升使用体验，也不会过早重构。
